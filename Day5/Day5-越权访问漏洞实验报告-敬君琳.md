## 三、实验原理

越权访问漏洞（又称IDOR，Insecure Direct Object Reference）是指Web应用程序在对用户请求进行处理时，没有对用户身份和所请求的资源之间进行充分的权限校验，导致攻击者可以通过修改请求中的参数（如URL中的user_id、表单中的隐藏字段等）来访问或操作本不属于自己的资源。

越权访问通常分为两类：

（1）水平越权：同一权限级别的用户之间相互访问对方的数据。例如，普通用户A通过修改URL中的user_id参数查看普通用户B的个人信息。

（2）垂直越权：低权限用户访问高权限用户的数据或功能。例如，普通用户通过修改URL参数查看管理员的信息。

本次实验在一套Flask用户管理系统上进行。该系统在个人中心（/profile）和充值（/recharge）两个路由中，故意省略了权限校验逻辑，user_id参数直接从URL或表单中获取，没有与当前登录用户的session进行比对，从而造成了越权访问漏洞。

## 四、实验内容与步骤

### 4.1 新增个人中心功能（/profile路由）

**步骤1**：在app.py中新增/profile路由，支持GET请求。该路由从URL参数获取user_id（如/profile?user_id=1），根据user_id从数据库查询用户资料，包括用户名、邮箱、手机和余额，并将查询结果显示在个人中心页面。路由代码中不验证当前登录用户和要查询的user_id是否匹配，这意味着任何人只要修改URL中的user_id参数，就可以查看任意用户的个人资料。

**步骤2**：新建templates/profile.html模板文件，继承base.html。页面展示用户信息表（ID、用户名、邮箱、手机、余额），同时包含充值表单（金额输入框、充值按钮），使用隐藏字段（`<input type="hidden">`）传递user_id。

**步骤3**：修改templates/base.html导航栏，在登录后的菜单中添加"个人中心"链接。

**步骤4**：修改templates/index.html欢迎页面，在已登录状态下添加"个人中心"快捷入口按钮。

**关键代码（/profile路由）**：

```python
@app.route("/profile")
def profile():
    user_id = request.args.get("user_id", "")
    # 注意：不从 session 获取 user_id，不验证当前用户是否有权查看
    profile_user = get_user_by_id(user_id)
    return render_template("profile.html", user=user_info, profile_user=profile_user)
```

### 4.2 新增充值功能（/recharge路由）

**步骤1**：在app.py中新增/recharge路由，支持POST请求。该路由从表单接收user_id和amount参数，直接修改用户数据中的余额字段（balance = balance + amount）。路由代码中不检查amount是否为负数，也不验证发起充值请求的用户是否有权操作对应的user_id。

**步骤2**：充值成功后重定向到/profile?user_id={user_id}页面，显示更新后的余额。

**关键代码（/recharge路由）**：

```python
@app.route("/recharge", methods=["POST"])
def recharge():
    user_id = request.form.get("user_id", "")
    amount = int(request.form.get("amount", "0"))
    # 注意：不检查 amount 是否为负数，不验证操作权限
    if user_id_int in user_balances:
        user_balances[user_id_int] = user_balances[user_id_int] + amount
    else:
        user_balances[user_id_int] = amount
    return redirect(f"/profile?user_id={user_id_int}")
```

### 4.3 越权访问漏洞验证

完成以上功能开发后，我对系统中的越权访问漏洞进行了验证测试。以下是测试过程：

**验证一：水平越权——普通用户查看其他普通用户资料**

- 以alice用户登录系统（密码：alice2025），进入个人中心页面。
- 在浏览器地址栏中，将URL从/profile?user_id=2修改为/profile?user_id=1。
- 页面成功显示了admin用户（user_id=1）的个人资料，包括邮箱（admin@example.com）、手机（13800138000）和余额（99999）。
- 这说明普通用户alice可以通过修改URL参数查看到另一个用户admin的信息，存在水平越权漏洞。

**验证二：垂直越权——普通用户查看管理员资料**

- 以alice用户登录系统后，直接访问/profile?user_id=1。
- alice作为普通用户（role: user），成功查看到admin管理员（role: admin）的全部个人资料。
- 这说明系统没有对角色权限进行校验，存在垂直越权漏洞。

**验证三：越权充值——为其他用户充值**

- 以alice用户登录系统后，访问/profile?user_id=1（admin的个人中心）。
- 在充值表单中输入金额100，点击充值按钮。
- 充值成功后，admin的余额从99999变为100099。而正常逻辑下，alice不应该能够为admin的账户充值。
- 这说明系统存在越权操作漏洞。

**验证四：负数金额充值**

- 以任意用户登录后，在充值表单中输入负数金额（如-500）。
- 提交后，用户的余额被减少了500。正常逻辑下，充值金额应该必须为正数。
- 这说明系统缺少对充值金额的合法性校验，可能导致恶意减少他人余额。

**验证五：Burp Suite抓包验证**

- 配置浏览器代理为127.0.0.1:8080，启动Burp Suite拦截请求。
- 在浏览器中访问/profile?user_id=2并提交充值请求。
- 在Burp Suite的Proxy标签页中拦截到POST /recharge请求，可以看到请求体中包含user_id=2和amount=100。
- 修改请求体中user_id的值为1，然后转发请求。
- admin用户的余额发生变化，而alice的余额不变。这说明通过Burp Suite修改请求参数可以实现越权操作，且这种攻击在前端是无法防御的。

## 五、漏洞分析

本次实验中的越权访问漏洞主要由以下几个原因造成：

**user_id参数从客户端获取且可被篡改**：在/profile路由中，user_id直接从URL查询参数（request.args）中获取，没有与session中存储的当前登录用户身份进行关联。攻击者只需修改URL中的user_id值即可查看任意用户的资料。正确的做法应该是从session中获取当前登录用户的身份信息，只允许用户查看自己的资料。

**缺少权限校验逻辑**：两个新增路由（/profile和/recharge）中完全没有权限检查代码。系统没有验证"发起请求的用户"和"要操作的目标用户"是否为同一人，也没有检查用户角色是否为管理员。这导致了水平越权和垂直越权两种漏洞。

**充值金额未做合法性校验**：/recharge路由中直接使用int()转换amount参数，没有检查金额是否为负数。攻击者可以提交负数金额来恶意减少任意用户的账户余额。

**敏感操作未要求二次认证**：充值属于敏感操作（涉及账户余额变动），但系统没有要求用户输入密码或进行其他形式的二次确认。这使得攻击者利用越权漏洞进行恶意充值时更加容易。

**隐藏字段可被篡改**：充值表单中使用`<input type="hidden">`传递user_id，这种方式只是让参数对普通用户不可见，但无法阻止攻击者通过浏览器开发者工具或Burp Suite修改该值。客户端的所有数据都应该被视为不可信的。

---

## 六、修复建议

1. **从Session获取用户身份**：user_id应从session中获取当前登录用户的身份信息，而不是从URL参数或表单参数中获取。例如：user_id = session.get("user_id")，这样攻击者就无法通过修改URL参数来查看其他人的资料。

2. **添加权限校验**：在/profile和/recharge路由中，应验证当前登录用户的身份。只有管理员或用户本人才允许查看和操作对应的资料。对于涉及角色权限的操作，还需要检查用户的角色（role）是否满足要求。

3. **校验充值金额**：充值金额应校验为正数（amount > 0），并进行合理的范围限制（如单笔充值不超过10000元），防止恶意输入。

4. **添加CSRF Token**：在充费等敏感操作的表单中添加CSRF Token，防止跨站请求伪造攻击。

5. **记录操作日志**：对充值等敏感操作记录日志，包括操作人、操作时间、目标用户、金额等信息，便于事后审计和追溯。

## 七、实验总结

通过本次实验，我深入理解了越权访问漏洞（IDOR）的原理和危害。在开发Web应用时，如果开发者过于信任客户端提交的参数，不在服务端进行严格的权限校验，就很容易产生越权访问漏洞。攻击者只需要修改URL中的user_id等参数，就可以查看甚至操作其他用户的数据，这在真实的网络攻击中是非常常见的。

本次实验中，我在已有的登录、注册、搜索、头像上传功能基础上，新增了个人中心和充值两个功能模块。在编写代码时，我按照要求故意没有添加权限校验，user_id从URL参数和表单参数中获取而不是从session获取，也没有校验充值金额的正负。通过这些有意留下的漏洞，我实际验证了水平越权（alice查看admin的资料）、垂直越权（普通用户查看管理员信息）、越权操作（为他人账户充值）以及负数金额充值等多种攻击场景。

在PortSwigger平台上的越权访问练习也帮助我理解了IDOR漏洞在实际Web应用中的表现形式。通过Burp Suite抓包和修改请求参数，我体会到客户端的任何校验都是不可靠的，安全校验必须在服务端进行。

这次实验让我认识到，安全开发的核心原则之一是"不要信任客户端输入"。所有来自客户端的参数（URL参数、表单数据、Cookie、HTTP头等）都可能被恶意篡改。开发者应该在服务端对每一次请求进行身份验证和权限校验，确保用户只能访问和操作自己被授权的资源。这对于今后从事网络安全工作具有重要的指导意义。
